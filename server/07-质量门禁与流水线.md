# 07 — 质量门禁与流水线

> 本文件定义"模块完成"与"系统通过"的精确机器化判据，以及调度脚本应如何串起所有 agent。
>
> 所有阈值与命令必须由调度脚本机械执行，不允许任何 agent 自行降低门槛。

---

## 1. 三道门禁

| 门禁 | 数值阈值 | 验证命令 |
|---|---|---|
| 测试全绿 | unit / property / component / e2e 四套 0 失败 | `pnpm test:<m>` && `pnpm test:e2e:<m>` |
| 覆盖率 | branch ≥ 90%、line ≥ 95%、function ≥ 95%、statement ≥ 95% | `pnpm vitest run --coverage tests/{unit,property,component}/<m>/**` |
| 变异 | kill rate ≥ 95%（存活率 < 5%） | `pnpm test:mutation:<m>` |

任意一项不达标 ⇒ 模块判定**未完成**，调度脚本必须回退对应 agent。

---

## 2. 模块完成清单（可机器抓取）

每个模块完成时，调度脚本检查：

```bash
# 每模块 7 个文件必存在
- src/server/routers/<m>.ts                        # （非 _app.ts）
- src/components/<m>/                              # 至少 1 个组件
- src/app/(app)/<m>/page.tsx                       # 列表页
- tests/unit/<m>/                                  # ≥ 1 个 .test.ts
- tests/property/<m>/  (M3, M4, M5 必有；其他可空)
- tests/component/<m>/                             # ≥ 1 个 .test.tsx
- tests/e2e/<m>.e2e.ts                             # 1 个 .e2e.ts

# 三道门禁全过
$ pnpm test:<m>          → exit 0
$ pnpm test:e2e:<m>       → exit 0
$ pnpm test:mutation:<m>  → exit 0
```

## 4. CI 流水线（GitHub Actions 模板）

`.github/workflows/ci.yml`：

```yaml
name: CI
on: [pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with: { node-version: 20.12.2, cache: pnpm }
      - run: pnpm install --frozen-lockfile

      - name: Type check
        run: pnpm tsc

      - name: Lint
        run: pnpm lint

      - name: DB schema (test)
        env: { DATABASE_URL: file:./prisma/test-ci.db, NODE_ENV: test }
        run: pnpm prisma db push --skip-generate

      - name: Unit + Property + Component
        env: { DATABASE_URL: file:./prisma/test-ci.db, NODE_ENV: test }
        run: pnpm test:coverage

      - name: Build (test)
        env: { NODE_ENV: test }
        run: pnpm build:test

      - name: Install Playwright
        run: pnpm exec playwright install --with-deps chromium

      - name: Start Next.js (test mode)
        env: { NODE_ENV: test, PORT: 3001, DATABASE_URL: file:./prisma/test-ci.db }
        run: pnpm start:test &

      - name: Wait for ready
        run: |
          for i in {1..30}; do
            curl -sf http://localhost:3001/api/test/reset -X POST && break
            sleep 1
          done

      - name: E2E
        env: { CI: 'true', E2E_BASE_URL: 'http://localhost:3001' }
        run: pnpm test:e2e

      - name: Mutation (full)
        run: pnpm test:mutation
```

> **顺序关键**：build:test 必须在启动 server 前完成；server 必须 ready 后再跑 E2E。

## 5. ship-module 流程（人类视角的"完成一个模块"）

```
1. spec-agent       写 server/specs/<m>.md（如尚未存在或需求变更）
2. trpc-contract-agent 写 src/server/routers/<m>.ts 骨架（如尚未存在）
3. <m>-e2e-test     写 tests/{unit,property,component,e2e}/<m>/**
   ⇒ 提交 PR：tests-only-<m>，人类审 → merge → git tag tests-frozen-<m>-v1
4. parallel:
     <m>-backend-impl   填 src/server/routers/<m>.ts
     <m>-frontend-impl  写 src/app/<m>/** + src/components/<m>/**
   循环跑 pnpm test:<m> 直至全绿
5. mutation-agent  scope=<m> 跑变异；不达标 → 退回 e2e-test 加严断言
6. integration-agent（最后）跑跨模块 E2E
   ⇒ 全部门禁绿 = 系统通过
```

> **冻结测试**：tests/<m>/** 在第 3 步 merge 后被 git 标记。任何后续对它的修改必须由人类提 PR；hook 拦截 impl-agent 的 Edit/Write。

## 6. 触发条件矩阵（agent 决定继续还是回退）

| 现象 | 判定 | 处理 |
|---|---|---|
| `pnpm test:unit:<m>` 红 | impl 未完成 | impl-agent 继续 |
| `pnpm test:e2e:<m>` 红 + unit 全绿 | 前端 / 端到端流断了 | frontend-impl-agent 修，必要时 backend-impl-agent 协助 |
| coverage < 阈值 | 漏测 | e2e-test-agent 补测 |
| mutation > 阈值 | 测试断言不严 | e2e-test-agent 加严断言 |
| 所有绿 + coverage + mutation 达标 | 模块完成 | 进入 integration |
| integration 红 | 跨模块行为破裂 | integration-agent 报 issue；具体是哪个模块由失败定位 |

## 7. 报告与产物

| 产物 | 位置 | 谁生成 |
|---|---|---|
| coverage HTML | `coverage/index.html` | vitest |
| mutation HTML | `reports/mutation/index.html` | stryker |
| playwright trace | `playwright-traces/<m>-<test>.zip` | e2e（仅失败时） |
| junit xml（CI） | `junit.xml` | vitest reporter |

`vitest.config.ts` 中加 reporter：

```ts
test: {
  reporters: process.env.CI ? ['default', ['junit', { outputFile: 'junit.xml' }]] : ['default'],
}
```

## 8. 升级 issue 渠道

当 impl-agent 认为某条测试错误（与 spec 矛盾、或物理上不可实现）时，**禁止自改测试**。规定动作：

1. 写 `inventory-app/issues/<timestamp>-<m>-<short>.md`
2. 内容含：测试文件路径、行号、测试名、与 spec 对比的疑点
3. 退出本轮 agent，提示调度脚本"已升级"
4. 调度脚本由人类审，决定改 spec 还是改测试，再走 ship-module 流

issue 模板：

```markdown
# Issue: <模块>-<简述>

- file: tests/<层>/<m>/<file>:<line>
- agent: <m>-backend-impl / <m>-frontend-impl
- date: YYYY-MM-DD HH:MM

## 现象
<描述测试断言>

## 与 spec 的冲突
<引用 server/specs/<m>.md 第 N 行>

## 推断
<这是 spec 错 / 测试错 / 测试与实现都对但理解不同>

## 建议
<如何修复>
```

## 9. 何时认为整个系统通过

```
所有 7 个模块各自：
  - test:<m> 全绿
  - test:e2e:<m> 全绿
  - coverage:<m> 达标
  - mutation:<m> 达标

且
  - test:e2e:integration 全绿
  - tsc --noEmit 0 错误
  - lint 0 错误
  - mutation 全量 kill ≥ 95%
```

调度脚本检查到上述条件即可宣告 ship。

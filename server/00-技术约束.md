## 4. 单号编码规则

| 单据 | 前缀 | 格式 | 示例 |
|---|---|---|---|
| 进货单 | `RH` | `RH` + `YYYYMMDD` + `0001`（4 位序列号，按 `purchaseDate` 当天计） | `RH202605020001` |
| 出货单 | `CH` | `CH` + `YYYYMMDD` + `0001`（4 位序列号，按 `salesDate` 当天计） | `CH202605020001` |

商品编码 `code`：正则 `^[A-Za-z0-9-]{3,20}$`，全局唯一。

## 5. 状态机

| 单据 | 字段 | 取值 | 流转 |
|---|---|---|---|
| 进货单 | `status` | `DRAFT` / `CONFIRMED` | `DRAFT → CONFIRMED`（不可逆） |
| 出货单 | `status` | `DRAFT` / `CONFIRMED` | `DRAFT → CONFIRMED`（不可逆） |

库存仅在 `confirm` 时变动；`DRAFT` 期间允许编辑/删除；`CONFIRMED` 后只读。

## 6. 错误码字面量（全局枚举）

所有 tRPC 抛出的 `TRPCError` 必须使用以下 `message` 之一，字面值精确匹配：

| message 字面量 | 适用 code | 含义 |
|---|---|---|
| `UNAUTHORIZED` | `UNAUTHORIZED` | 未登录或 token 失效 |
| `FORBIDDEN` | `FORBIDDEN` | 无权限 |
| `BAD_REQUEST` | `BAD_REQUEST` | Zod 校验抛出（仅在边界使用） |
| `CODE_EXISTS` | `CONFLICT` | 商品编码已存在 |
| `CATEGORY_NOT_FOUND` | `NOT_FOUND` | categoryId 不存在 |
| `SUPPLIER_NOT_FOUND` | `NOT_FOUND` | supplierId 不存在 |
| `PRODUCT_NOT_FOUND` | `NOT_FOUND` | productId 不存在 |
| `PRODUCT_HAS_STOCK` | `CONFLICT` | 库存 > 0 不能删除商品 |
| `CATEGORY_IN_USE` | `CONFLICT` | 分类被商品引用，不能删除 |
| `SUPPLIER_IN_USE` | `CONFLICT` | 供应商被商品/订单引用，不能删除 |
| `CATEGORY_NAME_EXISTS` | `CONFLICT` | 分类名重复 |
| `SUPPLIER_NAME_EXISTS` | `CONFLICT` | 供应商名重复 |
| `INSUFFICIENT_STOCK` | `CONFLICT` | 出货数量超过库存 |
| `EXCEEDS_MAX_STOCK` | `CONFLICT` | 进货后库存超过 maxStock |
| `ORDER_NOT_FOUND` | `NOT_FOUND` | 进货单或出货单不存在 |
| `ORDER_NOT_DRAFT` | `CONFLICT` | 仅 DRAFT 单可编辑/删除/确认 |
| `ORDER_ALREADY_CONFIRMED` | `CONFLICT` | 单据已确认，不能重复确认 |
| `INVALID_CREDENTIALS` | `UNAUTHORIZED` | 登录失败 |

> Zod 校验失败由 tRPC 直接转 `BAD_REQUEST` 并附 zodError，agent 无需手写。
>
> **HTTP 错误码不参与契约**：所有错误均通过 `TRPCError` 的 `message` 字面量识别。

## 13. 跨模块 import 白名单（hooks 强制）

`src/<moduleA>/` 严禁直接 import `src/<moduleB>/`，但以下例外允许：

- `src/server/db.ts`（Prisma 实例）
- `src/server/trpc.ts`（base router、procedure factory）
- `src/lib/**`（纯工具函数）
- `src/components/layout/**`（侧栏、顶栏、面包屑）
- `src/components/shared/**`（确认弹窗、空态、loading 等无业务的基础组件）
- `src/types/**`（共享类型，不含业务逻辑）
- `@prisma/client` 生成的类型

**特别地**：`src/server/routers/<moduleA>.ts` 严禁 import `src/server/routers/<moduleB>.ts`。如需跨模块业务，由 router 直接调用 Prisma 完成（事务在 router 内）。

## 14. 全局不变量（系统级）

- **G1** SQLite 单库；不引入 Redis / 队列。
- **G2** 业务时间戳由 Prisma 默认值生成（`@default(now())` / `@updatedAt`）；不允许应用层覆盖。
- **G3** 所有金额、单价用 JS `number`（IEEE-754）；持久化用 `Float`；前端显示保留 2 位。**约束：单价 ≤ 99999.99，金额 ≤ 999999999.99。**
- **G4** 任意时刻 `product.quantity ≥ 0`。
- **G5** 任意时刻 `product.minStock < product.maxStock`。
- **G6** 进货单 / 出货单一旦 `CONFIRMED` 永不可删除、永不可改。
- **G7** 单号在同一日内对每种单据类型严格单调递增。
- **G8** 删除商品当且仅当 `quantity == 0`。
- **G9** 删除分类当且仅当无 Product 引用。
- **G10** 删除供应商当且仅当无 Product 且无 Order（任一种）引用。
- **G11** 测试环境 `/api/test/reset` 与 `/api/test/seed` 仅在 `NODE_ENV === 'test'` 暴露；生产环境必须返回 404。
- **G12** 全部数值/字符串约束在 tRPC 入口（Zod）即拒绝；router 内部不重复校验，但**业务前置条件检查的顺序**由 spec 明确指定（违者门禁红）。

## 15. 不要做的事

- 不要引入 Redux / Zustand 等额外状态管理（用 React Query + 局部 useState 已足够）。
- 不要引入除 antd 外的 UI 库（不要 MUI、不要 Chakra、不要 shadcn）。
- 不要引入 ORM 之外的查询构造器。
- 不要写 RESTful 路由（仅 `/api/trpc/*` 与 `/api/test/*` 两条命名空间）。
- 不要在测试中 mock 数据库；必须使用真实 SQLite 临时文件。
- 不要在测试中 mock tRPC client；E2E 必须打真实 HTTP。
- 不要硬编码从 `process.env.DATABASE_URL` 之外的位置读 DB 路径。
- 不要把 BRD 里的"可选模块"（多仓库、邮件通知、订单作废）变成必做项；属于 **OUT OF SCOPE**。

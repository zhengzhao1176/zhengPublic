# 05 — 测试架构（4 层 + 双门禁）

> 本文档把 `AI开发测试框架.md` 的理论落到本项目。
> 浏览器自动化（E2E）层的具体写法在 `06-Playwright-CLI规范.md`。
>
> 严格遵循"信任契约"：tests/ 由 `<m>-e2e-test` 写、impl agent 不能改；测试通过即视为正确。

---

## 1. 金字塔（数字是单模块的目标量级）

```
                ┌──────────────┐
                │   E2E (4-6)   │   playwright-cli 浏览器自动化（持久化 profile）
              ┌─┴──────────────┴─┐
              │  Component (~12)  │   Testing Library + userEvent
            ┌─┴───────────────────┴─┐
            │   Property (3-6)       │   fast-check 不变量
          ┌─┴───────────────────────-─┴─┐
          │       Unit (~25)             │   Vitest + tRPC createCaller + 真 SQLite
          └─────────────────────────────-┘
                       ↑
            ┌──────────┴──────────┐
            │  Coverage 门禁       │   ≥ 90% 分支
            │  Mutation 门禁       │   < 5% 存活
            └─────────────────────┘
```

E2E 层独立运行（需启动 Next.js test 模式），其余三层在 Vitest 内。

```ts
import type { PrismaClient } from '@prisma/client'
import { appRouter } from '@/server/routers/_app'
import { createCallerFactory } from '@/server/trpc'

export type AnyCaller = ReturnType<ReturnType<typeof createCallerFactory<typeof appRouter>>>

export function makeCaller(db: PrismaClient, user: { id: number; username: string } | null = null): AnyCaller {
  return createCallerFactory(appRouter)({ db, user })
}

export function makeAuthedCaller(db: PrismaClient, userId = 1, username = 'admin') {
  return makeCaller(db, { id: userId, username })
}
```

## 4. Unit 测试规范

**目的**：验证 procedure 在固定输入下的行为。

**写法（举例）：**

```ts
// tests/unit/products/create.test.ts
import { describe, it, expect, beforeAll, beforeEach, afterAll } from 'vitest'
import type { PrismaClient } from '@prisma/client'
import { makeTestDb, resetAll, seedCategory, seedSupplier } from '@/tests/helpers/db'
import { makeAuthedCaller, type AnyCaller } from '@/tests/helpers/caller'

let db: PrismaClient
let caller: AnyCaller
let cat: { id: number }
let sup: { id: number }

const validInput = (overrides: Record<string, unknown> = {}) => ({
  code: 'P001',
  name: 'Widget',
  categoryId: cat.id,
  costPrice: 10.5,
  sellPrice: 20,
  quantity: 5,
  unit: '件',
  supplierId: sup.id,
  minStock: 1,
  maxStock: 100,
  ...overrides,
})

beforeAll(() => {
  db = makeTestDb()
  caller = makeAuthedCaller(db)
})

beforeEach(async () => {
  await resetAll(db)
  cat = await seedCategory(db)
  sup = await seedSupplier(db)
})

afterAll(async () => {
  await db.$disconnect()
})

describe('products.create — input', () => {
  it('rejects invalid code (regex)', async () => {
    await expect(caller.products.create(validInput({ code: 'AB' }))).rejects.toThrow()
    await expect(caller.products.create(validInput({ code: 'P 01' }))).rejects.toThrow()
  })
  // ... 其它输入约束 ...
})

describe('products.create — business rules', () => {
  it('throws CATEGORY_NOT_FOUND when categoryId unknown', async () => {
    await expect(caller.products.create(validInput({ categoryId: 99999 }))).rejects.toThrow(/CATEGORY_NOT_FOUND/)
  })
  it('throws SUPPLIER_NOT_FOUND when supplierId unknown', async () => {
    await expect(caller.products.create(validInput({ supplierId: 99999 }))).rejects.toThrow(/SUPPLIER_NOT_FOUND/)
  })
  it('throws CODE_EXISTS for duplicate code', async () => {
    await caller.products.create(validInput({ code: 'DUP' }))
    await expect(caller.products.create(validInput({ code: 'DUP', name: 'X' }))).rejects.toThrow(/CODE_EXISTS/)
  })
  it('checks category before supplier before code (precondition order)', async () => {
    await caller.products.create(validInput({ code: 'EXIST' }))
    await expect(
      caller.products.create({
        ...validInput(),
        code: 'EXIST',
        categoryId: 99999,
        supplierId: 99999,
      }),
    ).rejects.toThrow(/CATEGORY_NOT_FOUND/)
  })
})

describe('products.create — round trip', () => {
  it('persists exactly what was sent (numeric & trim)', async () => {
    const input = validInput({ name: '  Widget  ', costPrice: 99.99 })
    const created = await caller.products.create(input)
    const fetched = await caller.products.byId({ id: created.id })
    expect(fetched.name).toBe('Widget')
    expect(fetched.costPrice).toBe(99.99)
  })
})
```

**必须覆盖**（每个 procedure）：
- 每条 Zod 约束的拒绝用例
- 每个错误码（business rules）
- 至少 1 条 precondition order 测试
- 成功路径的持久化 round-trip

## 5. Property 测试规范

**目的**：随机输入打不变量；防止 impl-agent "凑数"绕过。

```ts
// tests/property/products/invariants.test.ts
import { describe, it, beforeAll, beforeEach, afterAll, expect } from 'vitest'
import fc from 'fast-check'
import { makeTestDb, resetAll, seedCategory, seedSupplier } from '@/tests/helpers/db'
import { makeAuthedCaller } from '@/tests/helpers/caller'

const codeArb = fc.stringMatching(/^[A-Za-z0-9-]{3,20}$/)
const moneyArb = fc.float({ min: Math.fround(0.01), max: 99999, noNaN: true, noDefaultInfinity: true })
  .map((n) => Math.round(n * 100) / 100)
  .filter((n) => n > 0)
const nameArb = fc.string({ minLength: 1, maxLength: 100 }).filter((s) => s.trim().length > 0)
const stockRangeArb = fc.tuple(fc.integer({ min: 1, max: 999_999 }), fc.integer({ min: 1, max: 1_000_000 }))
  .filter(([a, b]) => a < b)

const validInputArb = () => fc.record({
  code: codeArb,
  name: nameArb,
  costPrice: moneyArb,
  sellPrice: moneyArb,
  quantity: fc.integer({ min: 0, max: 1_000_000 }),
  unit: fc.string({ minLength: 1, maxLength: 20 }),
  stockRange: stockRangeArb,
})

let db, caller
beforeAll(() => { db = makeTestDb(); caller = makeAuthedCaller(db) })
afterAll(async () => { await db.$disconnect() })

describe('products invariants', () => {
  it('I2 round-trip: any valid create round-trips through byId unchanged', async () => {
    await fc.assert(fc.asyncProperty(validInputArb(), async (raw) => {
      await resetAll(db)
      const c = await seedCategory(db); const s = await seedSupplier(db)
      const input = {
        code: raw.code, name: raw.name, categoryId: c.id, supplierId: s.id,
        costPrice: raw.costPrice, sellPrice: raw.sellPrice,
        quantity: raw.quantity, unit: raw.unit,
        minStock: raw.stockRange[0], maxStock: raw.stockRange[1],
      }
      const created = await caller.products.create(input)
      const fetched = await caller.products.byId({ id: created.id })
      expect(fetched.code).toBe(input.code)
      expect(fetched.name).toBe(input.name.trim())
      expect(fetched.costPrice).toBe(input.costPrice)
      expect(fetched.minStock).toBeLessThan(fetched.maxStock)
    }), { numRuns: 30 })
  })

  it('I1 uniqueness anti-property: 2nd create with same code always rejects', async () => { /* ... */ })
})
```

**必须覆盖**（每个模块）：
- 每条 spec invariants（I*) ≥ 1 个测试，每个 ≥ 30 runs
- ≥ 1 个 anti-property（违反约束的输入必被拒）

## 6. Component 测试规范

**目的**：组件在隔离环境的交互、校验、错误展示。

```ts
// tests/component/products/product-form.test.tsx
import { describe, it, expect, vi } from 'vitest'
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { ProductForm } from '@/components/products/product-form'

const cats = [{ id: 1, name: '类别A' }]
const sups = [{ id: 1, name: '供应商A' }]

it('shows 编码必须为3-20位字母数字或短横线 when code invalid; blocks submit', async () => {
  const onSubmit = vi.fn()
  render(<ProductForm categories={cats} suppliers={sups} onSubmit={onSubmit} />)
  const user = userEvent.setup()
  await user.type(screen.getByLabelText('编码'), 'AB')
  await user.click(screen.getByRole('button', { name: /保存/ }))
  expect(await screen.findByText('编码必须为3-20位字母数字或短横线')).toBeInTheDocument()
  expect(onSubmit).not.toHaveBeenCalled()
})
```

**必须覆盖**（每个表单）：
- 每条字面错误文案的展示（`00-技术约束.md` §7.1）
- 每条 invalid case 都有 `expect(onSubmit).not.toHaveBeenCalled()`
- 提交中按钮 disabled + 文案变 `保存中...`
- 成功路径：`expect(onSubmit).toHaveBeenCalledWith(...)`
- 至少一条断言全部错误均带 `role="alert"`

## 7. E2E 测试规范

详见 `06-Playwright-CLI规范.md`。本节仅给出原则：

- E2E 在**真 Next.js（test 模式）+ 真 SQLite + 真 HTTP** 上跑，不允许任何 mock
- 每条 E2E 用 `/api/test/reset` + `/api/test/seed` 准备夹具
- 选择器优先 `getByLabel` / `getByRole`；表格行/操作按钮用 `data-testid`
- 必须有 ≥ 1 条 "客户端校验时网络请求 = 0" 断言（per module）

## 8. 覆盖率与变异门禁

### 8.1 覆盖率（v8）

```bash
pnpm vitest run --coverage tests/unit/<m>/** tests/property/<m>/** tests/component/<m>/**
```

阈值（`vitest.config.ts`）：

```ts
test: {
  coverage: {
    provider: 'v8',
    include: ['src/server/routers/<m>.ts', 'src/components/<m>/**', 'src/app/<m>/**', 'src/app/(auth)/login/**'],
    thresholds: {
      lines: 95,
      branches: 90,
      functions: 95,
      statements: 95,
      perFile: true,
    },
    reporter: ['text', 'html'],
  },
}
```

### 8.2 变异（Stryker）

`stryker.conf.json`：

```json
{
  "$schema": "./node_modules/@stryker-mutator/core/schema/stryker-schema.json",
  "packageManager": "pnpm",
  "testRunner": "vitest",
  "coverageAnalysis": "perTest",
  "checkers": ["typescript"],
  "tsconfigFile": "tsconfig.json",
  "mutate": [
    "src/server/routers/**/*.ts",
    "src/lib/order-no.ts",
    "!src/server/routers/_app.ts"
  ],
  "thresholds": { "high": 95, "low": 90, "break": 95 }
}
```

> 变异**仅覆盖核心业务**：tRPC routers + 单号生成。前端 UI 由 component + E2E 兜底，不进 stryker（避免代价过高）。
> `break: 95` 即变异 kill rate < 95% （= 存活率 > 5%）⇒ 流水线红。

### 8.3 模块通过判据

```bash
pnpm test:unit:<m>        # all green
pnpm test:property:<m>    # all green
pnpm test:component:<m>   # all green
pnpm test:e2e:<m>         # all green
pnpm vitest run --coverage 1>cov.log  # branches ≥ 90%
pnpm test:mutation:<m>    # mutation kill rate ≥ 95%
```

四条全过 ⇒ 模块判定完成。

## 9. 测试间隔离（必读）

- 每个 unit/property 测试前 `await resetAll(db)`，**不可**依赖前一个测试的状态
- 每条 E2E 用 `/api/test/reset` + `/api/test/seed` 重置
- Vitest concurrency：单文件内 `it` **不能 .concurrent**（数据库相互污染）；同模块跨文件 OK
- E2E 测试**串行**运行（共享 `.playwright-profile/` 状态）

`vitest.config.ts` 关键：

```ts
import { defineConfig } from 'vitest/config'
export default defineConfig({
  test: {
    pool: 'forks',
    poolOptions: { forks: { singleFork: true } },  // 串行避免 SQLite 锁
    setupFiles: [],
    environmentMatchGlobs: [
      ['tests/component/**', 'jsdom'],
    ],
  },
  resolve: {
    alias: { '@': new URL('./src', import.meta.url).pathname },
  },
})
```

## 10. 信任契约 — 给 test-agent 与 impl-agent

| 规则 | 谁来执行 |
|---|---|
| `<m>-e2e-test` 不读 `src/server/routers/<m>.ts` 与 `src/components/<m>/**` 与 `src/app/<m>/**` | hooks (PreToolUse Read) |
| `<m>-backend-impl` 与 `<m>-frontend-impl` 不写 `tests/{unit,property,component,e2e}/<m>/**` | hooks (PreToolUse Edit/Write) |
| 测试依赖 spec 的字面量（错误码 / 文案）；impl 必须复用同一字面量 | 由"全绿 + mutation < 5%"机器化 |
| 任何 agent 跨模块 import → 拒绝 | hooks（解析 import 语句） |
| impl agent 想改测试 → 升级 issue | hooks 拒绝写 + agent 自动起新 issue |

## 11. 反作弊速查

| 反作弊 | 本项目落地 |
|---|---|
| D1 测试与实现共谋 | 角色完全分离（test-agent / impl-agent），文件权限隔离 |
| D2 实现按测试输入凑数 | property test ≥ 30 random runs / invariant；anti-property |
| D3 空断言 / 漏分支 | Stryker 变异 kill ≥ 95%、覆盖率分支 ≥ 90% |
| D4 mock 走捷径 | 真实 SQLite + tRPC createCaller + 真浏览器 + 真 HTTP |

## 12. test-agent 验收清单（每模块）

- [ ] 单元覆盖每个 procedure 的输入约束、错误码、precondition order
- [ ] 属性测试 ≥ 1 个 invariants 测试 + ≥ 1 anti-property，每个 ≥ 30 runs
- [ ] 组件测试覆盖每条字段错误文案、`onSubmit` 未调用断言、提交中状态
- [ ] E2E 至少 1 条 happy path、1 条服务端错误、1 条客户端零请求
- [ ] 测试运行后**应该全红**（impl 还没填）；TS 编译 + import 通过
- [ ] tests/<m>/** 中 0 个 import 路径指向 `src/server/routers/<m>.ts` 或 `src/components/<m>/**` 或 `src/app/<m>/**`（除类型 import）
- [ ] 任何 spec 中 invariant 不能没对应测试
